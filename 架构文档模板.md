# 架构文档编写模板与踩坑复盘

> 目的：让后续项目的架构文档能直接驱动 Agent 从计划到编码全程零确认。
> 本文档基于 VPS 中转网关 M1-M6 的实际协作复盘。

---

## 一、本次踩的坑（按代价排序）

### 坑 1：技术栈未声明（代价：1 轮确认）

**现象**：架构文档目录结构用了 `.py` 后缀，但没有显式写"Python + FastAPI + uv"。Agent 必须停下来问语言、框架、包管理器。

**根因**：目录结构隐含了语言，但没有声明完整的工具链。

**修复**：模板新增第 0 节"技术栈声明"。

---

### 坑 2：上游 URL 拼接歧义（代价：M6 冒烟测试才发现）

**现象**：架构文档写"调用 OpenAI Chat 兼容模型服务"，没有说明 base_url 是否包含 `/v1`。实现时代码在 base_url 后拼接 `/v1/chat/completions`，导致请求路径变成 `/v1/v1/chat/completions`。

**根因**：外部 API 的 URL 构造规则属于协议细节，但文档没有给出精确的请求构造规则。

**修复**：模板新增"外部接口契约"小节，要求写出完整的 URL 构造规则和请求示例。

---

### 坑 3：数据模型字段名歧义（代价：3 次测试重写）

**现象**：架构文档描述了 `working_state` 包含"情绪摘要"，但没有给出精确字段名 `emotion_summary`。实现时用了 `emotion`，测试用了 `emotion_summary`，导致反复修改。

类似问题：`pending_items` vs `todos`，`created_at` vs `timestamp`。

**根因**：数据合同描述了语义，但没有给出精确的字段名和类型。

**修复**：模板要求每个数据模型给出完整的字段表（名称、类型、是否必填、默认值）。

---

### 坑 4：错误类型签名未定义（代价：2 次适配器重写）

**现象**：架构文档写了"读取失败抛出异常"，但没有定义异常类的构造函数签名。`SampleReadError` 实际需要 `(sample_type, reason, cause)`，但调用方先用了 `(message)`。`UpstreamError` 需要 `(status_code, message)`，但调用方先用了 `(message)`。

**根因**：错误类型是跨模块契约，但只描述了"抛异常"，没有给出类签名。

**修复**：模板要求错误类型给出完整的类定义伪代码。

---

### 坑 5：环境变量未列出（代价：自行推断 + 配置不完整）

**现象**：架构文档提到"配置来自环境变量"，但没有列出全部变量名、默认值、校验规则。实现时遗漏了 `ACTIVE_TURN_INTERVAL_SECONDS` 和 `ACTIVE_TURN_ENABLED`。

**根因**：环境变量是部署配置的契约，但散落在各模块描述中，没有集中清单。

**修复**：模板新增"环境变量配置表"小节。

---

### 坑 6：SQLite 表结构未给出（代价：实现后返工）

**现象**：架构文档描述了 Outbox 的行为（幂等、游标查询、limit clamp），但没有给出 DDL。实现时主键设计、唯一约束、索引都需要自行推断。

**根因**：数据库 schema 是持久化契约，但文档只描述了行为，没有给出结构。

**修复**：模板要求持久化层给出 DDL 或等价的结构定义。

---

### 坑 7：API 参数细节缺失（代价：测试边界值不确定）

**现象**：`GET /v1/outbox` 的 `limit` 参数 clamp 到 1-100，`after` 是整数游标。文档写了"游标查询"但没有给出参数名、类型、默认值、边界。

**根因**：API 契约只描述了语义，没有给出精确的参数定义。

**修复**：模板要求每个 API 端点给出完整的参数表。

---

### 坑 8：OpenAI 错误响应格式未定义（代价：实现后对照官方文档）

**现象**：文档写"返回 OpenAI 兼容错误格式"，但没有给出具体的 JSON 结构。实现时需要查阅 OpenAI 官方文档确认 `error.type`、`error.code`、`error.param` 字段。

**根因**："兼容某协议"不等于"给出协议结构"。

**修复**：模板要求引用外部协议时给出精确的请求/响应示例。

---

## 7. 外部接口契约 [必填]

### 7.x [端点名]

**方法**：`POST /v1/xxx`
**认证**：`Authorization: Bearer {token}`

**请求参数**：

| 参数名 | 位置 | 类型 | 必填 | 默认值 | 校验 | 说明 |
|--------|------|------|------|--------|------|------|
| model | body | str | 是 | — | 非空 | 模型名 |
| messages | body | array | 是 | — | 非空 | 消息列表 |
| stream | body | bool | 否 | false | 必须 false | 首版不支持 |

**请求示例**：
```json
{
  "model": "xxx",
  "messages": [{"role": "user", "content": "hello"}],
  "stream": false
}
```

**成功响应**（200）：
```json
{
  "id": "chatcmpl-xxx",
  "object": "chat.completion",
  "choices": [...]
}
```

**错误响应**：

| 状态码 | 场景 | 响应体 |
|--------|------|--------|
| 400 | 参数校验失败 | `{"error": {"message": "...", "type": "invalid_request_error"}}` |
| 401 | 鉴权失败 | `{"error": {"message": "...", "type": "authentication_error"}}` |

> 引用外部协议时（如"OpenAI 兼容"），必须给出本端点使用的精确字段。
"兼容"不等于"给 Agent 自行查阅文档"。

## 9. 持久化结构 [必填，如涉及]

### 9.x [表名]

```sql
CREATE TABLE IF NOT EXISTS xxx (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    trigger_id TEXT NOT NULL UNIQUE,   -- 幂等约束
    content TEXT NOT NULL,
    created_at TEXT NOT NULL DEFAULT (datetime('now')),
    seq INTEGER NOT NULL
);

CREATE INDEX IF NOT EXISTS idx_xxx_seq ON xxx(seq);
```

**约束**：

> DDL 是持久化契约。行为描述（幂等、游标）不足以替代结构定义。

```
## 三、模板 vs 本次文档对照

| 模板小节 | 本次文档是否有 | 缺失代价 | 严重度 |
|---------|-------------|---------|--------|
| 0. 技术栈声明 | 没有 | 1 轮确认 | 中 |
| 1. 首版边界 | 有，质量高 | — | — |
| 2. 核心术语 | 有 | — | — |
| 3. 总体架构 | 有，质量高 | — | — |
| 4. 目录结构 | 有 | — | — |
| 5. 数据合同（字段表） | 有描述，无字段表 | 3 次重写 | 高 |
| 6. 错误类型定义 | 没有 | 2 次重写 | 高 |
| 7. 外部接口契约（参数表） | 有描述，无参数表 | 边界值不确定 | 中 |
| 8. 外部服务调用契约 | 没有 | URL 拼接 bug | 高 |
| 9. 持久化结构（DDL） | 没有 | schema 返工 | 中 |
| 10. 环境变量配置表 | 没有 | 遗漏配置项 | 中 |
| 11. 模块定义 | 有，质量高 | — | — |
| 12. 验收基线 | 有，质量高 | — | — |
| 13. 建议实现顺序 | 有，质量高 | — | — |

**结论**：本次文档在架构设计层面（1-4, 11-13）质量很高，但在实现契约层面（5 字段表、6 错误类型、7-8 接口细节、9 DDL、10 环境变量）存在缺失，导致编码阶段需要反复确认和返工。

## 四、使用方法

1. 复制模板中的 markdown 代码块
2. 填写所有 `[必填]` 小节
3. `[选填]` 小节按需填写
4. 完成后与提示词模板配合使用：

```
# 项目：[项目名]

## 一、参考文件
| 文件 | 路径 | 用途 |
|------|------|------|
| 架构文档 | [路径] | 按模板填写，包含技术栈到验收基线全链路 |
| 经验文档 | [路径] | 协作策略 |
| 技能目录 | [路径] | superpowers |

## 二~七：[同提示词模板]
```

架构文档越完整，提示词可以越简短。当模板的全部必填小节都已填写时，提示词只需写：
"依据架构文档第 13 节，从 M1 开始展开编码。"
```
