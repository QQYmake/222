# 工程指令复盘与协作改进

> 基于《工程控制论》的指令缺漏分析与 Agent 应对策略。纯概念性文档，作为长期协作记忆运行。

---

## 一、用户指令缺漏模式与 Agent 应对

### 1.1 反馈滞后：纠错发生在交付之后

**缺漏描述：**

格式、质量、边界约束未在任务下达时声明，而是在产出后才追加修正。控制论中，这属于**反馈通道过长**——偏差信号在经过完整执行周期后才回到输入端，导致已投入的计算和劳动被废弃重做。

根因不是"忘了说"，而是将格式约束视为可发现项而非可指定项。在工程控制论中，任何影响输出的约束都应作为前馈信号在输入端注入，而非依赖输出端的反馈校正。

**Agent 应对：**

- 接收任务时，主动推断可能的格式约束（文档受众、抽象层级、是否含案例、语言风格），在执行前一次性列出确认清单
- 若用户未声明输出格式，Agent 在执行前补一份"输出规格"供确认，而非默认格式后交付

---

### 1.2 验收标准隐含：可观测变量缺失

**缺漏描述：**

"验证正常""完善""测试通过"等表述未定义可观测的判据。控制论要求系统输出必须是可观测变量——没有可观测量的系统无法实现有效反馈控制。当验收标准隐含在执行者脑中而非显式于指令中时，执行者只能猜测目标值，偏差概率随任务复杂度上升。

**Agent 应对：**

- 对每个含"验证""测试""确认"的指令，执行前补充量化判据：退出码、日志关键词、数据条数、文件存在性、响应状态码
- 将判据作为 checklist 附在执行计划中，逐项标注通过/失败

---

### 1.3 决策点延迟暴露：前馈控制不足

**缺漏描述：**

工程系统中必然存在需要人工裁决的决策点（技术选型、容错策略、数据格式取舍）。当这些决策点由执行者在编码过程中发现并提出，而非由指令者在架构阶段预判并集中处理时，控制论称之为**前馈缺失**——不确定性未在影响控制目标前被消除，而是在下游造成返工。

决策点延迟暴露的代价：编码中断、上下文切换、已写代码可能因决策结果而废弃。

**Agent 应对：**

- 输出架构或计划时，主动附加"待决策清单"，格式：决策点 / 选项 / 代价 / 建议
- 在架构审核环节，将决策清单作为强制交付物，确保用户一次性处理所有不确定性

---

### 1.4 控制周期过长：单指令耦合多任务

**缺漏描述：**

单条指令包含多个有依赖关系的子任务，中间无显式反馈点。控制论中，控制周期过长会导致偏差累积——如果早期子任务的产出有偏差，要到后续子任务甚至交付时才能发现，修正成本随耦合深度指数增长。

这不是"任务多"的问题，而是"反馈点少"的问题。同样数量的子任务，如果每个完成后都有反馈点，控制周期就缩短了。

**Agent 应对：**

- 收到多任务指令后，先输出拆分计划并标注反馈点（哪些步骤完成后需暂停等待确认）
- 对高风险步骤（架构设计、接口定义）强制插入反馈点，即使指令未要求

---

### 1.5 环境基线漂移：系统参数不稳定

**缺漏描述：**

指令中引用的资源路径、环境变量、依赖版本在会话间不一致，导致执行者需要重新发现环境状态。控制论中，这相当于**系统参数随时间漂移**——控制器基于过时的环境模型发出指令，导致执行偏差。

根因是环境状态未持久化记录，每次会话都从零开始重建环境认知。

**Agent 应对：**

- 维护环境基线表（资源名 / 路径 / 版本 / 上次验证时间），每次会话开始时校验
- 将环境变更（文件移动、依赖更新、路径变更）即时写入基线表

---

### 1.6 容错约束未显式声明：可靠性盲区

**缺漏描述：**

指令通常聚焦于"功能正确"，对"失败时如何降级""数据如何恢复""哪些组件是单点"缺乏显式要求。控制论将可靠性视为系统设计的固有维度而非附加项——如果容错需求不在输入端声明，执行者可能选择无冗余的方案，在故障发生时无降级路径。

**Agent 应对：**

- 在架构计划中主动加入"已知风险与约束"章节，标注单点故障、降级策略、数据恢复路径
- 即使当前不实现容错，也将其显式记录为"已知约束"，避免未来被遗忘

---

### 1.7 抽象层级跳跃：指令粒度波动

**缺漏描述：**

同一轮指令中，有时给出极具体的约束（文件路径、变量名、端口号），有时给出极抽象的要求（"依据某方法论"），缺乏中间层级的过渡。控制论中，这相当于**控制信号的分辨率不一致**——高分辨率信号和低分辨率信号在同一通道传输，执行者需要在两种模式间频繁切换，增加解读偏差。

**Agent 应对：**

- 识别指令的抽象层级，对高层抽象指令先补充中间层规格（模块边界、接口契约、数据流），再进入实现
- 对低层具体指令，反向检查是否与高层架构一致，防止局部优化破坏全局结构

## 二、对本指令自身的批判

本文档的生成指令本身同样存在上述缺漏：

1. **反馈滞后**：格式约束（不夸奖、不含案例、纯概念、长期记忆）在初版交付后才追加，导致初版作废重写。若在任务下达时即声明这些约束，可避免一轮完整执行周期的浪费。

2. **验收标准隐含**："简单总结"的"简单"程度、"纯粹概念性"的边界、文档长度预期——均未量化。执行者只能自行判断粒度。

3. **抽象层级跳跃**：指令同时包含高层目标（"作为通用长期记忆运行"）和低层约束（"不要出现具体案例"），但未说明中间层的组织结构（章节如何划分、分析维度如何选取）。

**结论：** 这一模式本身验证了"反馈滞后"和"验收标准隐含"是根深蒂固的指令习惯，需要 Agent 通过主动预判来补偿，而非依赖用户在每次任务中完整声明所有约束。

---

## 三、可继续保持的做法

以下做法符合控制论原则，建议保持，无需调整：

- **阶段分解**：将大目标拆为有序阶段，每阶段有明确输入和输出
- **架构审核闸门**：在设计阶段设置人工审核点，阻止未经验证的方案进入实现
- **验证驱动**：每个阶段结束时要求测试或验证，形成闭环
- **不确定时主动追问**：对不理解的概念要求解释而非猜测，符合"先消除不确定性再执行"
- **要求方法论依从**：指定执行方法论（如 TDD），减少执行过程中的随意性

## 五、实战复盘：增量工程实施中浮现的新缺漏

以下模式在多里程碑增量交付（M1→M6）的实战中暴露，未在初始指令中预见。

### 5.1 验证层次断裂：Mock 通过不等于集成通过

**缺漏描述：**

单元测试使用 Mock Transport 覆盖了请求构造、响应解析、错误映射的全部分支，全绿。但首次接入真实 API 时立即暴露 URL 路径拼接重复（base_url 已含 `/v1`，客户端又追加 `/v1/chat/completions`）。Mock 验证了"逻辑正确性"，但未验证"协议假设正确性"。

控制论解释：**验证系统的分辨率不足**。Mock 是一种降分辨率传感器——它只能探测控制器内部逻辑的一致性，无法探测控制器对外部环境的模型假设是否准确。当验证体系只有一种分辨率的传感器时，存在系统性盲区。

**Agent 应对：**

- 对涉及外部集成的模块，始终设置两层验证：单元层（Mock，验证逻辑）+ 冒烟层（真实端点，验证协议假设）
- 冒烟测试默认跳过（`skipUnless` 读取环境变量），有凭据时手动触发，不消耗日常 CI 资源
- 在计划阶段标注"协议假设清单"：URL 拼接规则、Header 格式、认证方式、响应字段映射——这些是 Mock 无法自动发现的

---

### 5.2 计划格式约束延迟：伪代码格式的未声明预期

**缺漏描述：**

用户要求实现计划"只写伪代码，标注清楚数据输入+输出+针对输入数据做的指令"。这一格式约束在计划开始编写前才声明。若 Agent 在接收"写实现计划"任务时主动预判计划格式（伪代码 vs 自然语言 vs 代码骨架 vs 混合），可避免一轮格式返工。

这是 1.1（反馈滞后）的一个具体变体：格式约束不是针对最终交付物，而是针对中间产物（计划文档）。Agent 通常预判最终交付物的格式，但忽略中间产物的格式。

**Agent 应对：**

- 对"计划""设计""方案"等中间产物，执行前主动预判格式维度：抽象层级（伪代码/自然语言/代码骨架）、包含要素（数据流图/接口签名/验收标准）、详细程度
- 输出一份格式规格供确认，而非默认一种格式后交付

---

### 5.3 凭据生命周期盲区：使用后未主动清理

**缺漏描述：**

真实 API 凭据写入 `.env` 用于冒烟测试。测试完成后，Agent 未主动提醒用户清理凭据。用户自行在服务侧删除 key 后，`.env` 文件中仍残留明文 key，直到用户追问才被发现并清除。

控制论解释：**系统的安全状态未被纳入可观测变量**。Agent 将"功能完成"作为终止条件，但凭据的"使用→清理"是一个完整的安全控制周期。验证体系只覆盖了功能维度，未覆盖安全维度。

**Agent 应对：**

- 引入临时凭据时，主动声明生命周期：用途、有效期、清理时机
- 任务完成时，将"凭据清理"作为交付 checklist 中的强制项，即使功能已验证通过
- 在环境基线表中标记临时凭据为"transient"状态，定期校验是否已超期

---

### 5.4 沙盒与生产环境的操作性差异

**缺漏描述：**

在沙盒中启动服务进行端到端测试时，遇到进程管理问题：后台启动、端口占用、进程残留、启动失败时的日志定位。这些操作性问题不影响代码正确性，但影响"能否在目标环境中运行"这一更基本的验收维度。

控制论解释：**系统的可操作性是独立的控制目标**。代码正确性（功能层）和环境可运行性（操作层）是两个正交维度。验证体系只覆盖了前者，导致"代码全绿但服务起不来"的盲区。

**Agent 应对：**

- 对需要部署到 VPS/容器/服务器环境的项目，在 M6 阶段强制执行"操作性冒烟"：从零启动服务、验证端口监听、确认日志输出、模拟重启后数据持久
- 将启动命令、端口、日志路径、健康检查方式作为交付物的一部分，而非仅交付代码

---

### 5.5 增量回归的基线累积：测试数量作为系统稳定性的可观测变量

**经验描述（正面）：**

每个里程碑完成后执行全量回归。测试数量增长轨迹：83 → 181 → 204 → 249 → 264。每个里程碑的新增测试不破坏已有测试。这一做法有效防止了增量开发中的回归缺陷。

控制论解释：**累积的回归测试构成了系统的稳定性基线**。测试数量和通过率是系统状态的可观测变量。当新变更导致基线下降时，偏差信号立即可见。这是反馈控制的有效实现。

**Agent 应对（保持）：**

- 每个里程碑结束时不只跑该里程碑的新测试，而是跑全量回归
- 将测试数量变化记录在里程碑验收报告中，作为系统演化的可追溯指标
