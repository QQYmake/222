# 工程控制论协作经验

> 基于《工程控制论》的 Agent 协作策略。纯概念性长期记忆，适用于所有工程协作场景。

---

## 一、控制论框架

工程控制论的核心是：**通过前馈和反馈通道，使系统输出趋近目标值**。Agent 协作是一个控制系统：

- **用户** = 控制目标设定者
- **Agent** = 控制器
- **代码/产物** = 受控对象
- **指令** = 输入信号
- **验收结果** = 输出信号

控制质量取决于：前馈是否充分（不确定性是否在执行前消除）、反馈是否及时（偏差是否在代价放大前被发现）、可观测变量是否完备（系统状态是否可测量）。

以下所有缺漏模式都是这三个维度的具体失效形式。

---

## 二、前馈缺失类

前馈控制要求在影响控制目标之前消除不确定性。以下模式都是前馈信号未在输入端注入的变体。

### 2.1 反馈滞后：纠错发生在交付之后

格式、质量、边界约束未在任务下达时声明，而是在产出后才追加修正。偏差信号在经过完整执行周期后才回到输入端，已投入的计算被废弃重做。

根因不是"忘了说"，而是将格式约束视为可发现项而非可指定项。任何影响输出的约束都应作为前馈信号在输入端注入。

**Agent 应对：**
- 接收任务时，主动推断可能的格式约束（文档受众、抽象层级、是否含案例、语言风格），在执行前一次性列出确认清单
- 若用户未声明输出格式，Agent 在执行前补一份"输出规格"供确认

### 2.2 决策点延迟暴露

工程系统中必然存在需要人工裁决的决策点（技术选型、容错策略、数据格式取舍）。当这些决策点在编码过程中才被发现，而非在架构阶段集中处理时，不确定性在下游造成返工。

**Agent 应对：**
- 输出架构或计划时，主动附加"待决策清单"：决策点 / 选项 / 代价 / 建议
- 在架构审核环节，将决策清单作为强制交付物

### 2.3 中间产物格式约束延迟

用户要求计划"只写伪代码"——这一约束在计划开始编写前才声明。Agent 通常预判最终交付物的格式，但忽略中间产物（计划、设计、方案）的格式。

这是 2.1 的变体：格式约束不是针对最终交付物，而是针对中间产物。

**Agent 应对：**
- 对"计划""设计""方案"等中间产物，执行前主动预判格式维度：抽象层级（伪代码/自然语言/代码骨架）、包含要素（数据流图/接口签名/验收标准）、详细程度
- 输出一份格式规格供确认

### 2.4 控制周期过长：单指令耦合多任务

单条指令包含多个有依赖关系的子任务，中间无显式反馈点。如果早期子任务的产出有偏差，要到后续子任务甚至交付时才能发现，修正成本随耦合深度指数增长。

这不是"任务多"的问题，而是"反馈点少"的问题。

**Agent 应对：**
- 收到多任务指令后，先输出拆分计划并标注反馈点
- 对高风险步骤（架构设计、接口定义）强制插入反馈点

## 三、可观测性缺失类

控制论要求系统输出必须是可观测变量。没有可观测量的系统无法实现有效反馈控制。

### 3.1 验收标准隐含

"验证正常""完善""测试通过"等表述未定义可观测的判据。执行者只能猜测目标值，偏差概率随任务复杂度上升。

**Agent 应对：**
- 对每个含"验证""测试""确认"的指令，执行前补充量化判据：退出码、日志关键词、数据条数、文件存在性、响应状态码
- 将判据作为 checklist 附在执行计划中

### 3.2 验证层次断裂：Mock 通过不等于集成通过

单元测试使用 Mock 覆盖了全部分支，全绿。但首次接入真实 API 时立即暴露协议假设错误（URL 拼接重复、Header 格式不符）。Mock 验证了"逻辑正确性"，但未验证"协议假设正确性"。

Mock 是一种降分辨率传感器——它只能探测控制器内部逻辑的一致性，无法探测控制器对外部环境的模型假设是否准确。当验证体系只有一种分辨率的传感器时，存在系统性盲区。

**Agent 应对：**
- 对涉及外部集成的模块，设置两层验证：单元层（Mock，验证逻辑）+ 冒烟层（真实端点，验证协议假设）
- 冒烟测试默认跳过，有凭据时手动触发
- 在计划阶段标注"协议假设清单"：URL 拼接规则、Header 格式、认证方式、响应字段映射

### 3.3 增量回归基线累积

每个里程碑完成后执行全量回归。测试数量增长轨迹构成系统稳定性的可观测变量。当新变更导致基线下降时，偏差信号立即可见。

**Agent 应对（保持）：**
- 每个里程碑结束时不只跑该里程碑的新测试，而是跑全量回归
- 将测试数量变化记录在里程碑验收报告中

## 四、系统参数与状态类

### 4.1 环境基线漂移

指令中引用的资源路径、环境变量、依赖版本在会话间不一致，导致执行者需要重新发现环境状态。控制器基于过时的环境模型发出指令，导致执行偏差。

**Agent 应对：**
- 维护环境基线表（资源名 / 路径 / 版本 / 上次验证时间），每次会话开始时校验
- 将环境变更即时写入基线表

### 4.2 凭据生命周期盲区

真实 API 凭据写入 `.env` 用于冒烟测试。测试完成后，Agent 未主动提醒清理。系统的安全状态未被纳入可观测变量——验证体系只覆盖了功能维度，未覆盖安全维度。

**Agent 应对：**
- 引入临时凭据时，主动声明生命周期：用途、有效期、清理时机
- 任务完成时，将"凭据清理"作为交付 checklist 中的强制项
- 在环境基线表中标记临时凭据为 transient 状态

### 4.3 沙盒与生产环境的操作性差异

在沙盒中启动服务进行端到端测试时，遇到进程管理、端口占用、日志定位等问题。这些操作性问题不影响代码正确性，但影响"能否在目标环境中运行"。

代码正确性（功能层）和环境可运行性（操作层）是两个正交维度。验证体系只覆盖前者，导致"代码全绿但服务起不来"的盲区。

**Agent 应对：**
- 对需要部署的项目，在交付前强制执行"操作性冒烟"：从零启动服务、验证端口监听、确认日志输出、模拟重启后数据持久
- 将启动命令、端口、日志路径、健康检查方式作为交付物的一部分

## 五、信号质量类

### 5.1 抽象层级跳跃：指令粒度波动

同一轮指令中，有时给出极具体的约束（文件路径、变量名），有时给出极抽象的要求（"依据某方法论"），缺乏中间层级的过渡。高分辨率信号和低分辨率信号在同一通道传输，执行者需要频繁切换模式。

**Agent 应对：**
- 识别指令的抽象层级，对高层抽象指令先补充中间层规格（模块边界、接口契约、数据流），再进入实现
- 对低层具体指令，反向检查是否与高层架构一致

### 5.2 容错约束未显式声明

指令通常聚焦于"功能正确"，对"失败时如何降级""数据如何恢复""哪些组件是单点"缺乏显式要求。如果容错需求不在输入端声明，执行者可能选择无冗余的方案。

**Agent 应对：**
- 在架构计划中主动加入"已知风险与约束"章节
- 即使当前不实现容错，也将其显式记录为"已知约束"

## 六、Agent 主动应对策略汇总

| # | 应对策略 | 补偿的缺漏 | 控制论维度 | 触发时机 |
|---|---------|-----------|-----------|---------|
| 1 | 执行前输出"输出规格"供确认 | 反馈滞后、验收标准隐含 | 前馈 | 接收任务时 |
| 2 | 补充量化验收判据 checklist | 验收标准隐含 | 可观测性 | 含"验证""测试"的指令 |
| 3 | 架构计划附"待决策清单" | 决策点延迟暴露 | 前馈 | 架构/计划输出时 |
| 4 | 多任务指令拆分并标注反馈点 | 控制周期过长 | 前馈 | 接收多任务指令时 |
| 5 | 维护并校验环境基线表 | 环境基线漂移 | 系统参数 | 每次会话开始 |
| 6 | 主动加入"已知风险与约束" | 容错约束未声明 | 前馈 | 架构/计划输出时 |
| 7 | 识别抽象层级，补充中间规格 | 抽象层级跳跃 | 信号质量 | 指令粒度波动时 |
| 8 | 外部集成模块设置 Mock + 冒烟双层验证 | 验证层次断裂 | 可观测性 | 涉及外部 API 集成时 |
| 9 | 中间产物主动预判格式规格 | 中间产物格式约束延迟 | 前馈 | 输出计划/设计/方案时 |
| 10 | 临时凭据声明生命周期并强制清理 | 凭据生命周期盲区 | 系统状态 | 引入临时凭据时 |
| 11 | 部署前强制执行操作性冒烟 | 沙盒与生产操作性差异 | 可观测性 | 最终交付或部署前 |
| 12 | 每里程碑全量回归并记录测试基线 | 增量回归基线累积 | 可观测性 | 每个里程碑结束时 |

以上策略无需用户指令触发，由 Agent 在对应时机主动执行。

## 七、可继续保持的做法

- **阶段分解**：将大目标拆为有序阶段，每阶段有明确输入和输出
- **架构审核闸门**：在设计阶段设置人工审核点，阻止未经验证的方案进入实现
- **验证驱动**：每个阶段结束时要求测试或验证，形成闭环
- **不确定时主动追问**：对不理解的概念要求解释而非猜测
- **要求方法论依从**：指定执行方法论（如 TDD），减少执行过程中的随意性
