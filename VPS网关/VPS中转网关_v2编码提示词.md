## 二、技能加载与流程

按以下顺序加载并执行技能：

### 阶段 1——启动与取证

```text
using-superpowers
  → 读取 v2 架构文档、现有项目和编码经验文档
  → 检查 pyproject.toml、uv.lock、测试目录和 Git 状态
  → 保存初始 git status --short、git diff --stat 和相关文件 diff 作为用户变更基线
  → 运行 v1 全量测试，记录真实测试基线
  → 检查现有真实 API 冒烟方式，但不输出凭据
```

本阶段必须产出：

- 当前测试通过数与失败数；
- 当前模块—文件映射；
- v1 与 v2 的差异清单；
- 环境基线：Python、FastAPI、httpx、SQLite、uv、pytest 版本；
- 用户既有未提交变更基线；
- 仅列会阻塞实现的决策点。

### 阶段 2——实现计划

```text
writing-plans
  → 严格按架构文档第 13 节 M0—M7 顺序
  → 每个 Task 只写伪代码
  → 每个 Task 标注输入、处理指令、输出、接口和测试
  → 标明用户回合/主动回合/长轮询的数据流
  → 不在计划中写完整实现代码
```

计划必须设置以下反馈点：

1. M0 完成：证明用户 LLM 等待不再阻塞 Outbox HTTP。
2. M3 完成：证明纯测试工具循环闭环。
3. M5 完成：证明主动唤醒串行、用户/主动回合可并行。
4. M6 完成：证明 Outbox commit 后唤醒长轮询。
5. M7 完成：全量回归、重启恢复；有凭据则真实工具冒烟，无凭据则记录 `BLOCKED_EXTERNAL_CREDENTIAL`。

### 阶段 3——执行

每个 Task 都执行：

```text
test-driven-development
  → 先写失败测试
  → 运行并确认按预期失败
  → 写最小实现
  → 运行新增测试直到通过

executing-plans
  → 按 M0—M7 顺序推进
  → 不跳过依赖里程碑
  → 每个里程碑结束运行全量回归

systematic-debugging（出现失败时）
  → 先读取错误、堆栈和相关数据流
  → 找到根因后做局部修复
  → 禁止为了让测试变绿而削弱架构不变量
```

### 阶段 4——完成前验证

```text
verification-before-completion
  → 运行全量回归
  → 逐项对照架构文档第 12 节
  → 执行真实 OpenAI Chat 工具调用冒烟
  → 执行服务启动、关闭、重启恢复冒烟
  → 检查凭据和日志无泄漏

requesting-code-review
  → 审查模块边界、接口一致性、并发隔离和测试遗漏
```

只有存在 Git 分支且用户要求集成时，才执行 `finishing-a-development-branch`。不要自行提交、推送或创建 PR。

### 本任务对技能默认流程的显式覆盖

以下是用户对本任务的直接要求，优先于技能中的通用默认流程：

- 不创建 git worktree；直接在用户填写的现有 v1 项目中增量修改。
- 不自动 commit、push 或创建 PR。
- `writing-plans` 只采用任务拆分与依赖分析；计划正文只写伪代码，不写完整实现代码。
- `executing-plans` 在当前会话连续执行 M0—M7，不要求切换独立会话。
- 计划保存后进行一次自动一致性检查；无 Critical/Important 问题即进入 M0，不等待人工批准。
- 有子代理/reviewer 时进行独立计划审查；没有时由当前 Agent 按架构第 12、13、15 节逐项自检。缺少子代理不构成阻断。
- 代码审查针对当前工作树 diff 与实际文件，不依赖提交 SHA。

## 三、技术栈与固定选择

| 维度 | 选择 |
|---|---|
| 开发方式 | 在现有 coding v1 中增量修改 |
| 语言 | Python；版本沿用 `pyproject.toml` |
| HTTP 框架 | FastAPI |
| 上游协议 | OpenAI Chat Completions |
| HTTP 客户端 | httpx；v2 模型调用使用异步客户端 |
| 数据库 | SQLite；每次操作短连接，不长期共享 connection |
| 包管理器 | uv |
| 测试框架 | pytest；沿用项目现有异步测试插件 |
| 方法 | TDD |
| 运行形态 | 单进程、单实例；禁止多 worker |
| 工具来源 | 仅 VPS ToolRegistry |
| 状态组件 | 保持 v1 Sample 只读；本轮不实现自动演化 |

不得未经批准更换框架、数据库、包管理器、协议或目录体系。

## 五、执行节奏

采用以下节奏：

- 连续执行 M0—M7。
- 每个里程碑完成后运行新增测试和全量回归，记录测试数量变化。
- 不为普通可逆实现细节暂停询问，采用架构文档默认值。
- 只有以下情况才停止并请求用户：
  1. 架构文档存在互相矛盾的合同；
  2. 真实 v1 代码使目标无法局部实现；
  3. 需要数据删除、凭据变更或不可逆外部操作；
  4. 必须更换固定技术栈。
- 状态组件不进入本轮，即使实现过程中发现可顺手加入。
- 计划文件保存并完成自动一致性检查后立即进入 M0，不等待人工确认。
- 保留用户已有未提交变更：不得撤销、覆盖、暂存或归因给本轮；无关变更绕开。只有目标文件存在无法安全合并的重叠时才停止。

## 七、架构执行约束

以下约束优先级高于局部实现便利：

### 7.1 并发边界

- 用户对话调用 LLM 时，允许一个主动回合并行调用另一个 LLM 请求。
- 两个主动回合永远串行；冲突任务必须 `expired`，不得排队补跑。
- 用户回合不得获取主动回合锁。
- 每个回合创建独立 TurnContext；messages、tool history、计数器不得共享。
- FastAPI lifespan 是 Scheduler 唯一启动与停止入口；构造函数不得启动后台任务。

### 7.2 工具边界

- 工具全部由 VPS 注册和执行。
- 前端 `tools`/`tool_choice` 不得创建或覆盖服务器工具。
- 同一批 tool_calls 按返回顺序执行，不并行。
- 工具失败不重试，转换成 tool message 回灌模型。
- 最多 5 轮、10 次、单工具 15 秒；达到上限后不再调用模型。
- `echo_test`、`delay_test`、`fail_test` 只能在测试配置启用。

### 7.3 唤醒边界

- 唤醒时区 Asia/Shanghai；允许时间 08:00—24:00。
- 最短间隔 20 分钟；每天最多实际启动 10 次。
- 不限制合法未来任务的最远期限。
- 任务到期时已有主动回合运行，直接 `expired`。
- 过期任务不等待、不补跑。
- 随机时间只抽取一次并持久化；扫描不得重新随机。

### 7.4 Outbox 边界

- SQLite 每次操作使用短连接；不得把同一个 connection 长期共享给 HTTP 与 Scheduler。
- 长轮询等待期间不持有 SQLite connection 或 transaction。
- 必须先 commit，再 notify。
- 同时只允许一个活动读取请求；第二个返回 `reader_busy`。
- 每次最多原子领取一条消息。
- v2 接受领取后网络断开可能丢展示；不得擅自加入 ACK/租约系统。

### 7.5 范围边界

- 不实现 Sample 自动写回、记忆整理或情绪演化。
- 不实现多设备优先级、前端工具、多实例部署。
- 不修改 v1 identity/optional Sample 降级规则。
- 不降低现有测试覆盖来适配新代码。

## 八、错误与可观测性要求

所有错误必须符合架构文档第 8 节，并记录可关联 ID：

- 用户回合使用 `request_id/turn_id`。
- 主动回合使用 `wake_id/turn_id`。
- 工具使用 `tool_call_id/turn_id`。
- Outbox 使用 `event_id/trigger_id`。

不得：

- 把工具内部堆栈回灌模型或返回前端；
- 把 API Key、完整 Sample 或完整对话写日志；
- 捕获异常后假装成功；
- 在写入失败时通知长轮询；
- 在工具/唤醒失败时自动无限重试。

## 九、测试与验收要求

以架构文档第 12 节为唯一验收基线。执行前生成完整“验收追踪矩阵”，不得用本提示词中的摘要替代任何一项：

```text
架构验收编号
  → 对应测试文件/测试名
  → 执行命令
  → 预期结果
  → 实际证据
```

第 12 节中的每一行都必须出现在矩阵中，包括生产隐藏测试工具、客户端工具注入、重复 ID、数据库连接释放、claimed 不重复返回、通知竞态、未 startup 不启动 Scheduler，以及关闭异步 HTTP Client。

验证分两层：

```text
单元/集成测试：Mock 上游，验证全部控制分支
真实冒烟：实际 OpenAI Chat 兼容端点，验证 get_server_time 工具闭环
```

每个里程碑结束后：

- 运行该里程碑新增测试；
- 运行全量回归；
- 记录通过数、失败数和耗时；
- 不以 README 中旧的“264 passed”替代当前真实执行结果。

如果启动基线本来存在失败：

- 所有新增 v2 测试必须通过；
- 相对启动基线不得新增失败或跳过；
- 上一条只比较 pytest 的 failed/skipped；有据可查的 `not_run_external` 不属于测试回归。
- 既有失败只记录，除非直接阻塞 v2，不扩大范围修复；
- 最终报告必须分别列出“既有失败”和“新增回归”。

## 十一、最终交付格式

完成后只交付可验证结果：

1. 修改文件清单，按 M0—M7 分组。
2. 最终数据流与架构文档是否一致。
3. 完整验收追踪矩阵：架构编号、测试/冒烟命令、预期与实际证据。
4. 每条验证命令原文、执行时间、退出码，以及 passed/failed/skipped/xfailed 数。
5. 启动基线与最终全量回归的差异。
6. 真实工具冒烟的脱敏记录，或 `BLOCKED_EXTERNAL_CREDENTIAL`。
7. 服务启动、关闭、重启恢复结果。
8. `git diff --stat`、初始变更基线与本轮增量范围；不要自动提交，不把用户原有修改归因给本轮。
9. 尚未实现的第 14 节扩展能力。
10. 已知风险和下一步建议。

不要用“应该可以”“理论上通过”代替实际命令输出。
