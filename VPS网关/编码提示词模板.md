# 全局编码提示词模板

> 本模板与《架构文档模板》配合使用。架构文档定义"做什么"，本模板定义"怎么协作"。
> 使用方式：复制本模板，填入 `[占位符]`，作为每个新项目的首轮提示词。

## 二、技能加载与流程

加载 superpowers-main 中的以下技能，按阶段顺序推进：

### 阶段 1 — 启动

```
using-superpowers
  → 确定本次技能组合
  → 读取架构文档 + 经验文档
  → 确认技术栈、项目位置、预览形态
  → 判断是否需要 git worktree 隔离
```

### 阶段 2 — 计划

```
writing-plans
  → 按架构文档第 13 节实现顺序
  → 每个 Task 只写伪代码
  → 标注：数据输入、数据输出、对输入数据做的指令
  → 标注：Task 间的数据流与接口关系
  → 不在计划文档中写完整实现代码
```

### 阶段 3 — 执行（每个里程碑循环）

```
test-driven-development
  → 每个 Task：先写测试，测试失败，再写实现
  → 测试通过后标记 Task 完成

executing-plans
  → 按计划顺序逐 Task 执行
  → 每个 Task 完成后运行增量测试

systematic-debugging（遇到失败时触发）
  → 先取证：读取错误信息、堆栈、日志
  → 再定位：找到根因
  → 后修复：最小改动修复
  → 验证：重跑测试确认修复

verification-before-completion（声称完成前强制触发）
  → 运行全量回归测试
  → 对照验收基线逐项确认
  → 证据先于断言：无测试证据不得声称完成
```

### 阶段 4 — 收尾

```
requesting-code-review
  → 提交前自检：代码风格、接口一致性、边界覆盖

finishing-a-development-branch
  → 合并 / PR / 清理工作区
  → 更新 AGENTS.md
```

### 可选技能（视任务复杂度加载）

| 场景 | 技能 | 触发条件 |
|------|------|---------|
| 无架构文档，从想法开始 | `brainstorming` | 阶段 2 之前插入 |
| 需要隔离工作区 | `using-git-worktrees` | 阶段 1 中判断后加载 |
| 有 2+ 独立 Task 可并行 | `dispatching-parallel-agents` | 阶段 3 中按需加载 |
| 需要子 Agent 执行独立 Task | `subagent-driven-development` | 阶段 3 中按需加载 |
| 收到反馈后处理 | `receiving-code-review` | 阶段 4 后插入 |
| 需要创建新技能 | `writing-skills` | 任意阶段按需加载 |

### 技能选择速查

| 你的情况 | 必选 | 可选跳过 |
|---------|------|---------|
| 有完整架构文档 | writing-plans, executing-plans, test-driven-development, verification-before-completion | brainstorming 可跳过 |
| 只有想法没有文档 | brainstorming → writing-plans → executing-plans | 不可跳过 |
| 单人单线推进 | 上述四项 + systematic-debugging | 并行/子 Agent 类可跳过 |

## 四、计划格式

- 实现计划只写伪代码
- 每个 Task 标注：
  - **数据输入**：来源、类型、约束
  - **数据输出**：类型、格式、约束
  - **对输入数据做的指令**：按步骤编号，每步一个原子操作
- 各 Task 之间的数据流与接口关系用图或表格标注
- 不在计划文档中写完整实现代码

## 七、安全管控

### 7.1 凭据安全

- 所有 key/token/secret 只写入 `.env`，不硬编码到代码
- `.env` 必须在 `.gitignore` 中，不进入版本控制
- `.env.example` 中只放空占位符，不含真实值
- 临时凭据使用后立即清除，并在交付时确认无残留
- 代码中搜索确认无真实 key 残留

### 7.2 输入校验

- 所有外部输入（HTTP body / query / header）必须经过类型校验和格式校验
- 拒绝不符合数据合同的请求，返回明确错误码
- 不信任客户端传入的路径、文件名、SQL 片段
- 对用户输入做边界检查：长度限制、字符集白名单、数值范围

### 7.3 鉴权

- 所有对外暴露的接口必须有鉴权层
- 鉴权失败返回标准错误格式，不泄露内部状态
- key 比较使用常量时间比较，防时序攻击

### 7.4 错误处理

- 错误响应不泄露堆栈信息、内部路径、数据库结构
- 对外错误格式统一，内部日志详细
- 外部服务异常不传播原始错误给客户端

### 7.5 依赖安全

- 锁定依赖版本（`uv.lock` / `pnpm-lock.yaml`）
- 不引入未审查的第三方包
- 依赖更新时检查 changelog

### 7.6 数据安全

- 持久化数据（数据库/文件）设置合理权限
- 敏感数据不写入明文日志
- 临时文件使用后清理

### 7.7 部署安全

- 服务以最小权限用户运行
- 只暴露必要端口
- 健康检查端点不泄露内部信息
- 日志文件设置轮转，不无限增长

## 填写指南

### 什么时候填什么

| 时机 | 需要填写的节 | 原因 |
|------|------------|------|
| 项目启动前 | 第一节（参考文件） | 三份文件是 Agent 认知项目的基础 |
| 第一次对话 | 第二节（技能）、第三节（技术栈） | 消除确认来回；技术栈全程不变 |
| 第一次对话 | 第四节（计划格式）、第五节（节奏） | 确定协作方式 |
| 涉及外部 API 时 | 第六节（凭据） | 声明"有"或"没有"，即使 key 还没给 |
| 每个项目必填 | 第七节（安全管控） | 逐条确认或按项目裁剪 |
| 每轮任务 | 第八节（任务） | 指定本轮做什么 |

### 不变量 vs 变量

**不变量（一次声明，全程生效）：**
- 技术栈、包管理器、测试框架
- 计划格式偏好
- 项目位置
- 预览/部署形态
- 安全管控基线

**变量（按轮次给出）：**
- 具体实现顺序（引用架构文档章节）
- 凭据（可后给，但提前声明有无）
- 每轮推进多少

### 核心原则

- 不变量前置，变量延迟
- 技术栈只在第一次对话声明一次
- 不要在提示词中写伪代码或数据结构（架构文档已包含）
- 引用文档章节比复制内容更有效
- 指定技能名 > 指定技能目录 > "用 superpowers"
- 安全管控逐条确认，不可整段跳过
